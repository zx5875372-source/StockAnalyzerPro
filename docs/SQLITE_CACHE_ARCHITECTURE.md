# SQLiteCache Architecture

Milestone 3 Sprint 6 defines the durable SQLite cache architecture for
StockAnalyzerPro.

This document is design only. It does not change provider implementations,
`modules/downloader.py`, `modules/analyzer.py`, SAP Score, backtest strategy, or
runtime cache behavior.

## 1. SQLiteCache Goals

`MemoryCache` proves that cached providers can avoid repeated provider calls
inside one Python process. `SQLiteCache` is the next step: durable cache across
CLI runs.

Goals:

- Persist provider payloads across process restarts.
- Reduce repeated yfinance requests during repeated watchlist scans.
- Preserve provider, symbol, data type, period, date range, fetch time,
  expiration time, payload hash, and payload content.
- Keep cache access behind `ICache` and `CachedDataProvider`.
- Make cache entries inspectable for debugging and future CLI cache management.
- Support safe expiration and manual invalidation without changing analyzer,
  scoring, or reporting logic.
- Prepare for future DuckDB or analytical storage without adding DuckDB now.

Non-goals:

- Do not implement SQLiteCache in this sprint.
- Do not replace `MemoryCache` yet.
- Do not add cache CLI commands yet.
- Do not add OpenBB, FinMind, Polygon, or DuckDB.
- Do not store formal point-in-time financial snapshots yet.

## 2. `cache_entries` Schema

SQLite should begin with one table that can store all cache payload types.

```sql
CREATE TABLE IF NOT EXISTS cache_entries (
    cache_key TEXT PRIMARY KEY,
    provider TEXT NOT NULL,
    symbol TEXT,
    data_type TEXT NOT NULL,
    period TEXT NOT NULL DEFAULT 'none',
    start_date TEXT,
    end_date TEXT,
    fetched_at TEXT NOT NULL,
    expires_at TEXT,
    payload_json TEXT NOT NULL,
    payload_hash TEXT NOT NULL,
    schema_version INTEGER NOT NULL DEFAULT 1
);

CREATE INDEX IF NOT EXISTS idx_cache_entries_lookup
ON cache_entries (
    provider,
    symbol,
    data_type,
    period,
    start_date,
    end_date
);

CREATE INDEX IF NOT EXISTS idx_cache_entries_expires_at
ON cache_entries (expires_at);

CREATE INDEX IF NOT EXISTS idx_cache_entries_data_type
ON cache_entries (data_type);
```

Column rules:

- `cache_key` uses the same canonical string generated by `CacheKey.to_string()`.
- `provider` is lowercase, for example `yahoo`.
- `symbol` is normalized, for example `2330.TW`; non-symbol cache types may use
  a logical identifier.
- `data_type` is a stable logical type such as `financials`, `price_history`,
  `snapshot`, `info`, or `universe`.
- `period` is `annual`, `quarterly`, `daily`, or `none`.
- `start_date` and `end_date` are ISO date strings for ranged data.
- `fetched_at` and `expires_at` are UTC ISO-8601 datetime strings.
- `expires_at = NULL` means the entry does not expire automatically.
- `schema_version` supports payload migrations.

## 3. `payload_json` / `payload_hash` Design

`payload_json` stores a canonical JSON representation of the cached payload.
`payload_hash` verifies that the stored JSON was not corrupted or partially
written.

Hash rules:

- Use SHA-256 from Python standard library.
- Hash the exact UTF-8 encoded canonical JSON string.
- Canonical JSON should use stable key ordering.
- Whitespace should be minimized to avoid equivalent payloads producing
  different hashes.

Recommended canonical dump:

```python
json.dumps(payload, ensure_ascii=False, sort_keys=True, separators=(",", ":"))
```

Validation rules:

- On `set`, serialize payload, compute hash, and store both.
- On `get`, recompute hash from `payload_json`.
- If recomputed hash differs from `payload_hash`, treat the entry as corrupted.
- Corrupted entries must not be returned to analyzer, report, scan, or backtest.

Payload envelope:

```json
{
  "type": "FinancialData",
  "schema_version": 1,
  "payload": {}
}
```

The envelope makes deserialization explicit and allows future migrations without
guessing payload type from cache metadata alone.

## 4. Datetime / TTL / `expires_at` Rules

All cache datetimes should be timezone-aware UTC values.

Storage format:

```text
YYYY-MM-DDTHH:MM:SS+00:00
```

Rules:

- `fetched_at` is set when provider data is successfully written to cache.
- `expires_at` is calculated from `fetched_at + ttl`.
- `expires_at = NULL` means permanent cache until manual invalidation.
- `get(key)` returns a value only when the entry exists, hash is valid, and the
  entry is not expired.
- `exists(key)` follows the same validity rules as `get(key)`.
- Expired entries should remain stored until cleanup, unless `get(key)` chooses
  to delete them eagerly.

TTL defaults:

| Data Type | TTL |
|---|---:|
| `info` | 24 hours |
| `financials` | 7 days |
| `balance_sheet` | 7 days |
| `cashflow` | 7 days |
| `price_history` | 1 day |
| `snapshot` | Permanent or manual refresh |

`SQLiteCache` should use the same TTL policy as `MemoryCache` and
`CachedDataProvider`.

## 5. Serialization / Deserialization Strategy

SQLite stores text. Provider payloads are Python objects. The cache needs an
explicit conversion boundary.

Serialization principles:

- Serialize normalized internal models, not raw yfinance objects.
- Store only JSON-compatible primitives: dict, list, str, int, float, bool, and
  null.
- Convert dataclasses with explicit helper functions.
- Convert pandas objects to stable table-like payloads.
- Preserve enough type metadata to rebuild the expected Python object.

Suggested module layout for Sprint 7:

```text
data_provider/cache/
  sqlite_cache.py
  serialization.py
```

Suggested serializer functions:

```python
def serialize_payload(value) -> dict:
    ...

def deserialize_payload(envelope: dict):
    ...
```

The serializer should fail closed:

- Unsupported type raises a clear cache serialization error.
- Failed deserialization is treated as corrupted cache.
- Corrupted cache is a cache miss and should be diagnosed.

## 6. Payload Type Storage

### `FinancialData`

`FinancialData` should be stored as an envelope containing the main object and
nested `FinancialPeriod` objects.

Shape:

```json
{
  "type": "FinancialData",
  "schema_version": 1,
  "payload": {
    "symbol": "2330.TW",
    "company_name": "...",
    "industry": "...",
    "sector": "...",
    "price": 0.0,
    "pe": 0.0,
    "pb": 0.0,
    "current": {},
    "previous": {},
    "missing_fields": [],
    "diagnostics": []
  }
}
```

Rules:

- Preserve `missing_fields` and `diagnostics`.
- Preserve `None` as JSON null.
- Rebuild as `FinancialData` and `FinancialPeriod`, not plain dicts.

### `PriceHistory`

`PriceHistory` should be stored as metadata plus rows.

Shape:

```json
{
  "type": "PriceHistory",
  "schema_version": 1,
  "payload": {
    "symbol": "2330.TW",
    "start": "2025-01-01",
    "end": "2025-12-31",
    "rows": [
      {
        "date": "2025-01-02",
        "open": 0.0,
        "high": 0.0,
        "low": 0.0,
        "close": 0.0,
        "adjusted_close": 0.0,
        "volume": 0.0
      }
    ]
  }
}
```

Rules:

- Date indexes become explicit `date` fields.
- Column names should be normalized to lowercase snake case.
- Missing OHLCV fields remain null.

### `dict`

Plain dict payloads can be stored directly inside the envelope.

Use cases:

- Company `info`
- Diagnostics summaries
- Snapshot metadata

Rules:

- Keys must be strings.
- Values must be JSON-compatible.
- Unsupported nested objects must raise serialization errors.

### `DataFrame`

Pandas `DataFrame` payloads should not be stored by pickle. They should be
converted to an explicit JSON table.

Shape:

```json
{
  "type": "DataFrame",
  "schema_version": 1,
  "payload": {
    "index_name": "date",
    "columns": ["date", "close", "volume"],
    "rows": [
      {"date": "2025-01-02", "close": 100.0, "volume": 1000.0}
    ]
  }
}
```

Rules:

- Convert datetime indexes to ISO strings.
- Convert NaN and pandas NA values to null.
- Preserve column order.
- Avoid pickle for security, portability, and inspectability.

## 7. Cache Corrupted Handling

Corruption can happen through manual edits, partial writes, disk issues, schema
drift, or serializer bugs.

Corruption signals:

- Invalid JSON in `payload_json`.
- `payload_hash` mismatch.
- Missing envelope fields.
- Unknown `type`.
- Unsupported `schema_version`.
- Deserializer raises an exception.

Handling rules:

- Do not return corrupted payloads.
- Record `cache_corrupted` diagnostic with cache key and reason.
- Treat corrupted entry as cache miss.
- Do not overwrite corrupted entry unless provider call succeeds.
- After successful provider call, replace corrupted entry with fresh payload.
- A future CLI command can list corrupted entries.

Optional eager invalidation:

- Sprint 7 may leave corrupted rows in place for debugging.
- Sprint 9 CLI can add `cache clean --corrupted` to remove them.

## 8. Cache Migration Strategy

`schema_version` is stored per entry to support future payload changes.

Migration rules:

- Version 1 is the initial JSON envelope design.
- Additive payload fields do not require immediate migration.
- Breaking payload changes require a new serializer version.
- SQLite table migrations should be explicit and idempotent.
- Migration scripts should run before cache reads.

Suggested metadata table for future use:

```sql
CREATE TABLE IF NOT EXISTS cache_metadata (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
```

Suggested metadata keys:

- `schema_version`
- `created_at`
- `last_migrated_at`

Initial Sprint 7 implementation can keep migrations minimal:

- Create tables if missing.
- Add missing indexes if missing.
- Refuse unknown future schema versions with a clear error.

## 9. Vacuum / Cleanup Expired Data Strategy

Expired cache rows should not immediately block reads because `get(key)` can
treat them as misses. Cleanup is still needed to control database size.

Cleanup strategies:

- Manual cleanup command in Sprint 9.
- Optional startup cleanup later, disabled by default.
- Delete expired rows where `expires_at IS NOT NULL AND expires_at < now`.
- Keep permanent snapshot rows where `expires_at IS NULL`.
- Run `VACUUM` only as an explicit maintenance operation.

Suggested SQL:

```sql
DELETE FROM cache_entries
WHERE expires_at IS NOT NULL
  AND expires_at < :now;
```

Vacuum rules:

- `VACUUM` can be expensive and locks the database.
- Do not run `VACUUM` on every scan.
- Run only from a future command such as:

```powershell
.venv\Scripts\python.exe app.py cache vacuum
```

or from a future dedicated CLI:

```powershell
.venv\Scripts\python.exe cache.py vacuum
```

## 10. Unit Test Plan

SQLiteCache tests should use temporary database files. Tests must not depend on
network calls.

Required tests:

- Creates schema:
  - New SQLite file starts empty.
  - Constructing SQLiteCache creates `cache_entries` and indexes.
- Cache hit:
  - Set a payload.
  - Create a new SQLiteCache instance pointing at the same file.
  - Get the same key and verify deserialized value.
- Cache miss:
  - Get unknown key returns `None`.
  - `exists(key)` returns false.
- Expired:
  - Set payload with short TTL.
  - Advance injected clock.
  - Assert `get(key)` returns `None`.
  - Assert row still exists until cleanup.
- Invalidate:
  - Set payload.
  - Invalidate key.
  - Assert row is removed or marked unavailable.
- Clear:
  - Set multiple keys.
  - Clear cache.
  - Assert all keys are unavailable.
- Hash validation:
  - Set payload.
  - Manually modify `payload_json`.
  - Assert get returns `None` and records `cache_corrupted`.
- Permanent snapshot:
  - Set snapshot with `ttl_seconds=None`.
  - Assert `expires_at` is null.
  - Assert it does not expire.
- Serializer coverage:
  - Round-trip `FinancialData`.
  - Round-trip `PriceHistory`.
  - Round-trip dict.
  - Round-trip DataFrame.

## 11. Migration Plan

Sprint 7: SQLiteCache Implementation

- Add `data_provider/cache/sqlite_cache.py`.
- Add `data_provider/cache/serialization.py`.
- Implement schema creation.
- Implement `get`, `set`, `exists`, `invalidate`, and `clear`.
- Add payload hash validation.
- Add unit tests with temporary SQLite files.
- Do not connect SQLiteCache to ProviderFactory by default yet.

Sprint 8: CachedDataProvider supports SQLiteCache

- Allow `ProviderFactory` to create cached providers with SQLiteCache.
- Keep MemoryCache as the default until SQLite behavior is fully verified.
- Add tests proving cached provider can use either MemoryCache or SQLiteCache.
- Add config hook for cache path.
- Do not change Analyzer or SAP Score.

Sprint 9: CLI cache management

- Add user-facing cache management commands.
- Minimum commands:
  - cache status
  - cache clear
  - cache invalidate provider/symbol/data type
  - cache cleanup expired
  - cache vacuum
- Make CLI output explicit about database path, row counts, expired rows, and
  corrupted rows.
- Keep destructive commands explicit and confirmed where appropriate.
